# DESIGN

## Модель данных

**`approval_requests`** — центральная сущность.
- `workspace_id`, `source_type` (`publication|scenario|edit|external`), `source_id`,
  `reviewer_user_ids` — только идентификаторы внешних сущностей, без их содержимого.
- `status`: `pending → {approved, rejected, cancelled}`. Последние три —
  финальные, обратных переходов из них нет (см. `app/core/state_machine.py`).
- `decided_by_user_id`, `decided_at`, `decision_note` заполняются при переходе
  в финальное состояние (`decision_note` — это `comment` из approve или
  `reason` из reject/cancel).

**`audit_log_entries`** — неизменяемый след действий: кто (`actor_user_id`),
что (`action`: created/approved/rejected/cancelled) и когда сделал с конкретной
заявкой. Пишется в той же транзакции, что и изменение самой заявки.

**`outbox_events`** — транзакционный outbox для будущей интеграции с другими
сервисами через события (`approval_request.created/approved/rejected/cancelled`).
Событие фиксируется в той же БД-транзакции, что и смена статуса — это гарантирует
consistency (либо оба изменения применились, либо ни одно), без прямого вызова
внешних систем из этого сервиса. Отдельный воркер-publisher, читающий
неопубликованные записи и отправляющий их в брокер, вынесен за рамки задания.

**`idempotency_records`** — результат первого выполнения мутирующего запроса
по ключу `(workspace_id, idempotency_key, endpoint)`, с fingerprint тела
запроса для обнаружения конфликтующего повторного использования ключа.

## Границы сервиса

Сервис не знает ничего о содержимом публикаций/сценариев/материалов и не
хранит данные пользователей, кроме идентификаторов. Он отвечает только за:
жизненный цикл заявки на согласование и её аудит. Все остальные сервисы
(публикации, пользователи, workspace) считаются внешними — их идентификаторы
принимаются как есть, без валидации существования (в реальной системе это
можно добавить через синхронный вызов/событие-подтверждение, но выходит за
рамки MVP).

## Изоляция workspace

`workspace_id` есть в каждой таблице и в каждом запросе к БД (WHERE
`workspace_id = :ws`). Дополнительно: `workspace_id` из URL всегда сверяется
с `workspace_id` из auth-контекста в dependency `require_action` — при
несовпадении возвращается `404` (не `403`), чтобы не подтверждать сам факт
существования чужого workspace/заявки (enumeration-защита).

## Обработка повторов (идемпотентность)

Два независимых механизма, решающих разные проблемы:

1. **Idempotency-Key (опциональный заголовок)** — защищает от повторной
   отправки одного и того же HTTP-запроса (двойной клик, ретрай сети).
   Ключ уникален в рамках `(workspace_id, endpoint)`; повтор с тем же телом
   возвращает сохранённый ответ без повторного выполнения операции; повтор
   с другим телом — `409`.
2. **Машина состояний** — защищает от повторного *осмысленного* действия:
   даже без Idempotency-Key нельзя перевести уже решённую (финальную) заявку
   в другое финальное состояние — второй `approve`/`reject`/`cancel` вернёт
   `409`.

Компромисс: если клиент не передал `Idempotency-Key`, `create` без ключа
может создать дубль при повторной отправке — это осознанно оставлено на
совести клиента (см. README), т.к. сервис не может отличить "два одинаковых,
но разных по смыслу запроса" без явного ключа.

## События / интеграции

Реализован Transactional Outbox (см. выше), а не прямая публикация в брокер,
по двум причинам:
- нет гарантии at-least-once без outbox (иначе можно потерять событие при
  падении между записью в БД и отправкой в брокер);
- сервис должен быть независим от конкретного брокера на этапе MVP — сам
  паблишер (Kafka/SQS/RabbitMQ-адаптер) добавляется отдельно, не меняя
  доменную логику.

Все payload'ы событий и audit log проходят через `sanitize_for_audit`
(allow-list полей) — так что даже при добавлении нового поля в модель оно не
попадёт в событие/лог, пока не будет явно разрешено.

## Логи и секреты

- Middleware логирует только `method`, `path`, `status_code` — не тело
  запроса/ответа.
- В моделях нет полей под токены, email, storage keys, signed URL или сырые
  provider-payload — их и не должно появляться, так как сервис работает
  только с идентификаторами и продуктовыми полями (title/description/note).

## Известные компромиссы / что не сделано

- `list` не использует cursor-пагинацию (только `limit/offset`) — для теста
  этого достаточно, для больших объёмов потребуется keyset-пагинация.
- Нет реального paбlisher'а для outbox (только таблица) — воркер и адаптер
  брокера не реализованы.
- Нет проверки существования `sourceId`/`reviewerUserIds` во внешних сервисах
  — идентификаторы принимаются как есть.
- Auth — заголовки без подписи/проверки (заглушка, что и требовалось заданием).
- Конкурентные решения по одной заявке (два approve одновременно) полагаются
  на atomarность транзакции БД и повторную проверку статуса в рамках неё;
  под очень высокой конкуренцией стоило бы добавить `SELECT ... FOR UPDATE`
  или optimistic locking по `updated_at`/версии — сейчас этого нет.
